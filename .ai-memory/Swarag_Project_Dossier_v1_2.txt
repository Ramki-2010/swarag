=========================================================
1. WHAT IS SWARAG
=========================================================

Swarag identifies Carnatic ragas from audio using a deterministic
DSP pipeline (not deep learning). It extracts pitch features and
matches them against known raga signatures.

The goal is to codify Carnatic musical grammar into computation.
Raga identity is structured grammar, not statistical coincidence.

Current: 7 ragas modeled (70 clips), 3 more staged.
Target: 72 Melakartas, janya ragas, live inference, Android app.

=========================================================
2. PIPELINE
=========================================================

Audio -> Vocal Isolation (Demucs/stems) -> Pitch Extraction (pYIN)
-> Tonic Estimation -> Normalization (cents) -> 72-bin PCD +
Directional Dyads -> IDF x Variance Scoring -> Guardrails -> Output

Key parameters:
  - 72-bin PCD (17 cents per bin)
  - Directional dyads with ALPHA=0.01 Laplace smoothing
  - IDF x Variance weighted dot-product
  - PCD_WEIGHT=0.8, DYAD_WEIGHT=0.2 (global, applied uniformly)
  - PER_RAGA_WEIGHTS: none -- Bhairavi 0.5/0.5 override retired v1.3.2
  - MIN_CLIPS_PER_RAGA=5 guardrail
  - MAX_DURATION_SEC=360 (6-minute cap)

Details: .ai-memory/architecture.md, .ai-memory/adr.md

7. KEY DECISIONS
=========================================================

- Deterministic DSP (small dataset, explainability, control)
- 72-bin PCD (17 cents/bin resolves microtonal swaras)
- IDF x Variance weighting (downweights common swaras)
- PCD-heavy weights 0.8/0.2, applied uniformly to all ragas (v1.3.2)
- Directional dyads (Carnatic ragas are asymmetric)
- MIN_CLIPS_PER_RAGA=5 guardrail (thin-data ragas poison models)
- Sandbox-first workflow (never edit production untested)
- UNKNOWN over wrong certainty (margin guardrails)
- Per-raga weight override for Bhairavi -- TESTED AND RETIRED (v1.3.2).
  Was believed validated; canonical rerun showed 0% decided. See ADR-006/013.
- Honest baselines over inflated numbers (L-041) -- reinforced hard by the
  67.4% fabrication discovery; see Section 11.
- Per-raga weight overrides are a trade for most ragas (L-042). No
  currently-active exception remains (Bhairavi was believed to be one;
  it was not).
- LOO numbers are only trusted when produced by a checked-in script
  (sandbox_loo_v131_canonical.py), never a hand-typed table (new, v1.3.2)

8. RISKS
=========================================================

- Abhogi/Kalyani parent-janya overlap makes PCD-only separation impossible
- Bhairavi model weak (14% LOO) -- needs diverse clips
- Mohanam model barely commits (9/10 UNKNOWN) -- needs diverse clips
- Kamboji only 3 real clips after Harikambhoji cleanup
- Harikambhoji/Kamboji parent-janya confusion can recur with other ragas
- No absolute OOD score floor (margin-only detection)
- Vocal-isolation mandate is policy-only, not enforced in code (BUG-009)
- Silent failures: except-Exception hides crashes as UNKNOWN
- Constants must stay synchronized across scripts -- concretely recurred
  2026-07-11: confusion_matrix_audit.py hardcoded the retired Bhairavi
  override as "canonical" one commit after recognize_raga_v12.py retired
  it. Fixed, but demonstrates the risk is not merely theoretical (L-002).
- Documentation can silently diverge from code and from itself even with
  an active audit process -- occurred twice in one week (2026-07-10 and
  2026-07-11 audits each found new staleness the prior pass missed)

11. DOCUMENTATION INTEGRITY (new section, 2026-07-11)
=========================================================

A hand-typed LOO table claiming 67.4% accuracy was treated as canonical
from 2026-03-31 until 2026-07-10, when an audit found its per-raga rows
summed to 34c/13w/23u -- not the stated 29c/14w/27u. It was never a real
run. It had directly motivated the Bhairavi 0.5/0.5 override (ADR-006),
which the real rerun then showed was actively harmful.

Standing rules going forward:
  1. Any LOO number treated as canonical must trace to a checked-in
     script (sandbox_loo_v131_canonical.py), not a hand-typed table.
  2. Before trusting a logged table, sum its own per-raga rows and check
     against its own TOTAL row. See .ai-memory/evaluation-protocol.md
     Section 4.
  3. docs/ARCHITECTURE.md is pure structural documentation (how/why the
     pipeline works) and holds no run data -- no LOO tables, no accuracy
     percentages, no per-raga results. All logged data (canonical numbers,
     per-raga tables, run IDs) lives only in .ai-memory/architecture.md
     and .ai-memory/datasets.md. The 2026-07-10 and 2026-07-11 corruptions
     of docs/ARCHITECTURE.md both trace to a results table being pasted
     into this file -- the file that should structurally never contain one.
  4. When a scoring-config decision (weight, override) is retired,
     search the ENTIRE repo -- docs and scripts -- for the old config,
     not just the file it was made in. The retirement in commit 21da815
     updated 2 docs and 1 script correctly but missed: docs/ARCHITECTURE.md
     (corrupted separately), PROJECT_STATUS.md, README.md, datasets/README.md,
     agent_spec.md, DEVELOPMENT.md, CONTRIBUTING.md, debug-playbook.md,
     this Dossier, and confusion_matrix_audit.py and sandbox_abhogi_ratio.py
     (two live scripts that would have silently used the retired config
     on their next run). All fixed 2026-07-11.
